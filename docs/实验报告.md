## 二、数据结构设计说明

### 2.1 图 ADT 设计概览

项目采用**分层模块化**架构，自上而下分为四层：

```
┌─────────────────────────────────────────────┐
│  main.cpp                                   │  ← 入口层：创建 CommandProcessor
├─────────────────────────────────────────────┤
│  CommandProcessor                           │  ← 命令层：命令 → 调用图操作 / 算法
├────────────────────┬────────────────────────┤
│  GAlgorithm        │  CsvIO                 │  ← 算法 & IO 层
│  (BFS/Dijkstra/    │  (ReadPlaces/          │
│   Prim/Tarjan/     │   ReadRoads/           │
│   K-Path)          │   WritePlaces/         │
│                    │   WriteRoads)          │
├────────────────────┴────────────────────────┤
│  LGraph                                     │  ← 图存储层
│  private:                                   │
│    v_map  vertices      顶点表               |
│    adj_map adj          嵌套邻接表           │
│    cat_map category_idx 类别倒排索引         │
│    int edge_count       逻辑边计数器         │
└─────────────────────────────────────────────┘
```

`LGraph` 向 `GraphAlgorithm` 层暴露只读遍历接口（`AllPlaceIds()`、`AllEdges()`、`GetAdjacentEdges()`），

算法模块不提供直接访问 `private` 索引实例的方法，除非强行`#define private public`/`篡改头文件`。增删改通过`LGraph`内部的公共方法操作，同时在内部维护索引一致



### 2.2 内部存储结构


```cpp
class LGraph {
private:
    using v_map  = std::unordered_map<std::string, LocationInfo>;
    using adj_map = std::unordered_map<std::string,
                    std::unordered_map<std::string, EdgeNode>>;
    using cat_map = std::unordered_map<std::string, std::vector<std::string>>;

    bool     directed;         // 是否是有向图（本项目始终为 false）
    v_map    vertices;         // 顶点表：place_id → LocationInfo
    adj_map  adj;              // 邻接表：from_id → (to_id → EdgeNode)
    cat_map  category_index;   // 类别索引：category → [place_id, ...]
    int      edge_count = 0;   // 逻辑边数（无向图不重复计数）
};
```

**哈希表结构示意：**

```
v_map 顶点哈希 :
        place_id 地点名  ----> Vertex 顶点
            即Locationinfo loc = v_map[place_id](实际使用find()获取迭代器)
            知地名查顶点

adj_map 邻接哈希 :
        from_id 起点 ----> pair<to_id, Edge>[] 终点与边的二元组哈希数组(即map)
            即unordered_map<string, EdgeNode> adj = adj_map[from_id]
            知顶点查邻接
        
        (from_id 起点 ----> to_id 终点) ----> EdgeNode 边
            或EdgeNODE edge = adj_map[from_id][to_id]
            知始末定位边

cat_map 类名哈希 :
        cat_id 类型名 ----> string[] 地点名数组
            即vector<string> = cat_map[cat_id]
            知类名查顶点数组
```

**无向图的双向存储**：每条无向边在 `adj` 中存两份记录（`adj[u][v]` 和 `adj[v][u]`），但 `edge_count` 只计 1

 `GetAdjacentEdges(u)` 无需判断方向，直接用 `adj[u]` 即可定位邻接，Dijkstra/Prim/BFS 的邻接查询都是 O(degree)。

## 数据结构的详细思考过程

### 1. 邻接矩阵？

#### 传统矩阵的空间爆满和繁琐扩容

邻接矩阵最显著的问题就是它 **O(V²)** 的空间复杂度

注意到必做数据中有一个 `|V| = 10000` 的稀疏拓扑，这意味着需要分配一个 **1e8** 的矩阵空间，这**极其浪费堆空间**

更严重的是，假如要 **增减顶点**，矩阵的 **行与列会频繁改变**，**空间重分配** 的开销是极大的

#### 可能的解决方法：稀疏矩阵 ADT

如果一定要用矩阵存储，面对这种大型图，一种可行的思路是另设计一个 **稀疏矩阵 ADT**

它在一定程度上解决了空间浪费，而且不论是课堂中提到的 **有序/无序三元组** 还是 **十字链** 形式，它们的追加操作复杂度均优于常规矩阵

但上面提到的各种形式，增删查改大多只停留在 **O(log N)** 的复杂度。链表的增删看似迅速，但那是对于已定位的情况而言的——**查找的 O(N) 复杂度不可忽略**。

当然，我们可以用 **哈希表** 设计稀疏矩阵 ADT 来大大降低这些操作的时间复杂度。但为了一个 ADT 去设计另一个 ADT 已经很繁琐，而且这还不如直接维护索引

#### 结论

综合上述论证，**邻接矩阵方案完全淘汰**。

---

### 2. 邻接表？

#### 邻接表的优势：`vector<list>` 形式

邻接表的空间复杂度无疑是极为优秀的 **O(V+E)**，可以说一点空间都没有浪费，在空间复杂度上算得上一骑绝尘

直接在内部构建的邻接表也很方便 DFS、BFS 等 **遍历算法**，甚至可以新提供一个接口，访问 graph 的邻接表，这相比其他要自行构建邻接表的方案便利了不少

对于这种形式的邻接表，增、查、删边的时间复杂度 **O(degree)**，对需要查邻接的算法友好

#### 邻接表的致命缺陷

为了实现从地点名到顶点的快速查找，显然且最优的解是维护一个 **string 到 vector 索引的映射**。但是依赖于索引就在增删操作中留下了大问题：

- 在增删顶点中，vector 需要大量的 **动态扩容**；
- 在 **删除** 顶点后，由于索引不能随意变更，要么造成一片 **无用的空洞邻接**，要么需要 **大量的重排**；
- 假如留下的空间或重排的是 `int`，那开销也勉强接受，但移动的数据结构是大量冗杂的 **字符串**。

#### 可能的解决方向：从 `vector` 到 `list` 再到 `unordered_map`

如果一定要解决这个问题，一个可行的思路是使用 **`list<list>`**，这样在删除的时候可以直接操作指针，一定程度上减少开销

但这样在查顶点的时候又会增加一个 **平均 O(V/2)** 的时间复杂度，进一步解决需要维护一个 **string 到 iterator 的映射**

这给我们提供了一个初步灵感，构造 **`unordered_map<list>`**，它不仅快，而且内存友好

#### 结论

如果一定要采用邻接表，**最好的选择是 `unordered_map<list>`**，但这个数据结构仍然不如双哈希，可以作为 **备选方案**

---

### 3. 边表？

#### 边表的优势

边表的优势屈指可数：除了高效的 **O(1) 追加**，就是这些数据结构中 **最低的空间复杂度 O(E)** ，这种 **以时间换空间** 的思路在这一项目中劣势很大

#### 边表的低效查找

无论是邻接矩阵还是邻接表，至少它们查邻接的效率尚可。但是边表查邻接需要遍历整个数组，时间复杂度 **O(E)**，这对本项目算法极不友好

甚至它对顶点的增删改也需要先同样遍历整个数组，还未考虑索引映射的维护，边表就已经 **极度不可接受**

可能的优化方法是使用分块数组查找，维护一个个bucket，但是这个其实已经是哈希的思路了，不必大费周章给边表打补丁

#### 结论

边表和邻接矩阵一样，**甚至更差**，不可接受

---

### 4. 嵌套哈希表（双哈希）

#### 双哈希的繁多优势

- **地名是字符串**，双哈希直接以字符串为键，**无需维护额外的 string→索引映射**，避免了删除顶点后产生的索引空洞或全局重排,解决传统邻接表痛点

- **增删边和判断边是否存在**  **O(1)** 的时间复杂度，动态扩容和判断存在极为方便，删边查边O(1)快于unordered_map<list>

- **删除顶点时**，双哈希只需遍历所有顶点O(V)清理邻接，邻接表需扫描所有边，邻接矩阵要做重分配——**在稠密图中优势尽显**

- **查邻接操作十分迅速**，对本项目算法友好


#### 结论

**嵌套哈希表是极其优秀的数据结构**，最终选择它作为实现方案

---



#### 最终结论

| 方案 | 查邻接 | 增删边 | 增删顶点 | 空间 | 采用? |
|------|--------|--------|---------|------|------|
| **邻接矩阵** `vector<vector<int>>` | O(1) | O(1) | O(V²) 重分配 | O(V²) | ❌ 顶点数不确定，字符串 id 需额外映射 |
| **普通邻接表** `vector<list<EdgeNode>>` | O(degree) | O(degree) 查找 + O(1) 插入/删除 | O(V+E) 需维护 id→索引；删除顶点后索引空洞 | O(V+E) | ❌ 需维护 id→索引映射；删除顶点后索引空洞 |
| **边表** `vector<EdgeNode>` + 可选索引 | O(E) 扫描（或 O(degree) 若建索引） | O(E) 查找 + 移动元素（或 O(degree) 若索引） | O(E) 扫描删除相关边 | O(E) | ❌ 邻接查询太慢，动态更新代价高 |
| **链表套链表** `list<list<EdgeNode>>` | O(degree) + 需额外映射定位顶点 | O(degree) 查找 + O(1) 删除/插入 | 增加 O(1)；删除需 O(E) 清理入边 | O(V+E) | ❌ 需维护 `map<string, iterator>` 定位顶点；删边/查边 O(degree)；删顶点 O(E) |
| **哈希套链表** `unordered_map<list<EdgeNode>>` | O(degree) | 增边 O(1)；删边/查边 O(degree) | 增加 O(1)；删除需 O(E) 清理入边 | O(V+E) | ❌ 查边/删边仍为 O(degree)；删顶点 O(E) |
| **嵌套哈希（双哈希）** `unordered_map<unordered_map<...>>` | O(degree) | O(1) 均摊 | 增加 O(1)；删除需 O(V) 清理入边 | O(V+E) × 2^(!directed) | ✅ 字符串 id 原生支持；边操作 O(1)；删顶点 O(V)；代码简洁 |

---


**选择嵌套 `unordered_map` 的核心理由：**

1. **place_id 是字符串**,无法直接作为数组下标。`unordered_map<string, ...>` 天然提供id→数据映射，省去维护 `id→int` 索引层的复杂度。
2. **图是动态的**,顶点和边会频繁增删。`unordered_map` 的插入/删除是 O(1) 均摊，不需要像 `vector` 那样处理扩容、移位或空洞回收。
3. **双向存储解决邻接查询**,无向图中 `adj[u]` 直接返回 u 的所有邻居，Dijkstra/Prim/BFS 的核心循环只需一次 `find(u)`（O(1)），然后遍历邻接（O(degree)）
4. **方便内部函数实现**,值得一提的是,在LGraph中自行提供了`方向性反转`的接口，这一数据结构能够很方便实现这一函数

**代价**：无向图每条边存两份，空间翻倍；AllEdges() 需 `from < to` 去重。对于最大规模（V=10000, E=15000），双存约 30000 条 EdgeNode × ~100 字节 ≈ 3MB，完全可接受。

## 三、关键算法说明与复杂度分析

### 3.1 连通分量分析（BFS）

- **时间复杂度**：O(V + E)
- **空间复杂度**：O(V)
- **实现要点**:

#### 有向图处理

本算法**不支持有向图**，如遇到有向图，会立即报错，返回空结构体

这是因为有向图的强连通分量不可以用朴素遍历方式计算，需要用到`Tarjan`扎口袋算法求强连通分量

假如只是想求有向图底图的连通分量，LGraph中提供了`ReverseDirection()`方法，可以将其转化为无向底图再应用此算法，考虑到接口简洁性，不再单独设立函数

#### 调用接口
| 方法 | 说明 |
|------|------|
|`IsDirected`|`有向图处理`|
| `GetAllPlaceIds` | `获取所有顶点信息` |
| `GetAdjacentEdges` | `获取所有邻接的open边` |

#### 算法要点
1. **利用`std::unordered_set`来确定顶点是否已经被走过**

这是因为到本项目仿照课程中使用`visited 布尔数组`需要维护额外映射，而且此方法也是`均摊O(1)`的时间复杂度，开销差异不大

1. **改造BFS而非改造DFS**

因为DFS需要在函数内部再定义一个`std::function`或`仿函数`实现递归（`lambda递归`也可以，但需要绑定自身，很繁琐），不如BFS简洁明了

DFS的实现需要同时将`size`信息作为参数参与递归，这降低了代码可读性，不如BFS可读性强

同时DFS容易在数据规模大时发生`栈溢出`，不过这不是主要问题

BFS依赖的`std::queue`的存储数据分配在堆上，堆空间更大，且在`RAII`机制下资源能得到释放，只要不是极端数据，不容易发生`堆溢出`

BFS核心：
```cpp
while (!q.empty()) {
  std::string cur = q.front();
  q.pop();
  ++size;

  //自行修改后的GetAdjacentEdges函数默认只提供open边
  std::vector<EdgeNode> neighbors = graph.GetAdjacentEdges(cur);
  for (const auto &e : neighbors) {
      const std::string &nb = e.to_id;
      //走过的顶点加入集合
      if (visited.find(nb) == visited.end()) {
          visited.insert(nb);
          q.push(nb);
      }
  }
}
```

#### 算法实现
1. 检验`有向性`，如果有向直接返回
2. 获取顶点信息以初始化`visited`集合，初始化`component_size`数组
3. 对于每个不在`visited`中的顶点做`BFS`，`BFS`的同时记录连通分量大小`size`，当某个顶点的`BFS`结束时，将`size`追加到`component_size`中
4. 所有顶点均遍历后，对`component_size`做降序排序，整理并返回结果


### 3.2 最短路径（Dijkstra）

- **时间复杂度**：O((V + E) log V)  
- **空间复杂度**：O(V)  
- **实现要点**：

#### 图的方向性

本算法**不区分有向图与无向图**，只要底层邻接关系正确即可正确运行

#### 调用接口
| 方法 | 说明 |
|------|------|
| `ExistVertex` | 检验起/终点是否存在于图中 |
| `GetAdjacentEdges` | 获取当前顶点的所有 open 邻接边（自动过滤 closed 道路） |


#### 算法要点

1. **使用小顶堆优化**  
   使用 `std::priority_queue` 存储 `(当前最短距离, 顶点ID)`的二元组，并传入 `std::greater<P>` 使其按距离升序排列，从而每次在 O(log V) 时间内取出当前未处理顶点中距离最小的一个。

2. **懒惰删除替代显式 visited 集合**  
   经过与AI的交互，发现弹出堆顶元素 `(d, u)` 时，若 `d != dist[u]`，直接 `continue` 忽略，便可以避免显式维护 `visited` 集合。此技巧代码简洁且不增加额外空间。

3. **权重参数化**  
   通过枚举类 `PathMode` 的值：
   - `PathMode::DIST` 使用 `e.distance`
   - `PathMode::TIME` 使用 `e.walk_time`  
   这相比于直接传入`Boolean`类型更加直观地表达了此参数的意思，十分优雅

4. **提前终止**  
   经过与AI的交互，发现一旦从堆中弹出目标顶点 `to_id`，即可直接 `break`
   
   因为 Dijkstra 保证第一次弹出时已经找到从起点到该顶点的全局最短路径，后续扩展不再必要，显著减少无效计算。

#### 算法实现步骤
1. 检查起/终点是否存在于图中，不存在则返回不可达结果
2. 初始化距离表 `dist`、前驱表 `prev` 和小顶堆 `pq`，将起点入堆，距离置零
3. 主循环：取出堆中距离最小的顶点，执行懒惰删除；若为目标顶点则提前终止；否则遍历其所有 open 邻接边，尝试松弛邻居顶点
4. 循环结束后，若终点距离为无穷大则返回不可达；否则从终点出发回溯整条路径并反转
5. 返回 `PathResult`（总代价、路径序列、可达标志）

### 3.3 时刻约束最短路径

- **时间复杂度**：O((V + E) log V)  
- **空间复杂度**：O(V)  
- **实现要点**：

#### 调用接口
| 方法 | 说明 |
|------|------|
| `ExistTimeVertex` | 检验符合当前时间的起/终点是否存在于图中 |
| `GetAdjacentEdges` | 获取当前顶点的所有 open 邻接边（自动过滤 closed 道路） |

#### 代码复用
本算法几乎与2.2 B 最短路径算法一致，故关键信息不再重复，唯二的两处改动如下

```cpp
// 起点或终点不存在,或开放时间不符,判断不可达
if (!graph.ExistTimeVertex(from_id, time) ||
    !graph.ExistTimeVertex(to_id, time)) {
    return PathResult();
}
```

```cpp
auto edges = graph.GetAdjacentEdges(u, true);
for (const auto &e : edges) {
    const std::string &v = e.to_id;
    // 邻居顶点必须在给定时刻开放
    if (!graph.ExistTimeVertex(v, time)) continue;
    ...}
```
`ExistTimeVertex`和`TimeValidator`以及`ValidCheck`方法负责维护，此处只需简单调用

### 3.4 必经点路径规划

- **时间复杂度**：O(k · (V + E) log V)，k = 必经点个数  
- **空间复杂度**：O(k · V)（每段均需临时距离表，但由于分段独立执行，实际峰值空间为一次 Dijkstra 的 O(V)）  
- **实现要点**：
  
#### 调用接口
**与最短路径算法完全一致**
| 方法 | 说明 |
|------|------|
| `ExistVertex` | 检验起/终点是否存在于图中 |
| `GetAdjacentEdges` | 获取当前顶点的所有 open 邻接边（自动过滤 closed 道路） |


#### 复用分段最短路径

本函数**不重复造轮子**，将必经点序列拆解为多段连续的序列，每一段直接调用已实现的 `GetShortestPath`，从而继承其全部特性

由于此最短路径的分路径也必然是最短路径（容易反证），这个算法的正确性得到保障

#### 重复节点的处理

相邻段之间存在天然的重复节点：第 `i` 段的终点正是第 `i+1` 段的起点。在拼接完整路径时，必须分类讨论：

- **第一段**（起点 → 第一个必经点）：直接将整条路径 `seg.path` 赋予 `full_path`。
- **后续段**（必经点₍ᵢ₎ → 必经点₍ᵢ₊₁₎，或最后一个必经点 → 终点）：跳过该段路径的第一个元素（即前一段的终点），将其余元素追加到 `full_path` 末尾。

具体实现为：
```cpp
if (i == 0) {
    full_path = std::move(seg.path);
} else {
    full_path.insert(full_path.end(),
                     seg.path.begin() + 1,   // 跳过重复的起点
                     seg.path.end());
}
```


#### 算法实现步骤

1. 构造完整节点序列 `stops`：起点 + 必经点列表 + 终点。
2. 依次对相邻的两个节点调用 `GetShortestPath`，得到段结果。
3. 若任一段不可达，立即返回不可达结果。
4. 累加总代价，并按规则拼接路径（首段全保留，后续段去头）。
5. 返回包含总代价、完整路径和可达标志的 `PathResult`。

### 3.5 最小生成树（Prim）

- **时间复杂度**：O((V + E) log V)  
- **空间复杂度**：O(V + E)（堆中最多存储 E 条边，visited 集合 O(V)）  
- **实现要点**：

#### 图的方向性

本算法**仅支持无向图**。若 `IsDirected()` 为真，直接抛出异常。最小生成树是定义在无向连通图上的概念，有向图的最小树形图Arborescence需要完全不同的算法


#### 调用接口
| 方法 | 说明 |
|------|------|
| `IsDirected` | `有向图处理` |
| `GetAllPlaceIds` | `获取所有顶点信息` |
| `GetAdjacentEdges` | `获取所有邻接的open边` |


#### 堆中冗余边的惰性删除

Prim 在将新顶点 `to` 加入 MST 后，会遍历其所有邻接边并入堆，这些边中可能包含指向**已加入 MST 的顶点**的边

算法在弹出堆顶元素时检查 `if (visited.count(to)) continue;`，若目标顶点已在树中则直接丢弃该边。这种策略避免了显式从堆中删除过时边的复杂操作


#### 自定义比较器

堆中元素为自定义结构体 `EdgeInfo`，按 `weight` 升序；权重相同时再按 `from`、`to` 的字典序次第比较。这保证了严格弱序，使得在面对多条等权边时选边规则确定、可复现，便于调试和回归测试

#### 连通性检查

当堆为空但已选取的边数仍不足 `|V| - 1` 时，说明存在孤立顶点或图不连通，此时函数返回空 `vector`

#### 算法实现步骤
1. 检查有向性，无向图继续。
2. 调用 `AllPlaceIds` 获取全部顶点，空图直接返回。
3. 任选一顶点作为起始，加入 `visited`，将其所有 open 邻接边按权重入堆。
4. 循环：弹出最小权重边，若目标顶点未访问则加入 MST，并将该目标的未访问邻接边入堆。
5. 当 MST 边数达到 `|V|-1` 或堆空时结束。
6. 若边数不足 `|V|-1`，返回空；否则返回 MST 边集。

#### *设计细节：Prim 而非 Kruskal*

校园导航图理论上**偏向稀疏图**，`Kruskal`在面对这种数据上理论上**具有一定的微弱优势**

但本项目仍然采用了`Prim`算法，主要原因在我设计的数据结构下，应用`Kruskal`需要维护**额外的映射层**：

- 图用 `std::string` 标识顶点，而 `DSU` 只能操作整数索引；因此 `Kruskal` 必须额外构建 `unordered_map<string, int>` 将地点名映射到 0..V-1
- 这样的操作不仅繁琐，它实质上在算法层重新发明了一套编号系统，这种行为非必要不推荐（但在后面的Tarjan中必须使用）

- 而`Prim` 直接使用 `GetAdjacentEdges(u, true)` 按需获取局部邻接边，配合 `unordered_set<string>` 记录已访问顶点，与Dijkstra风格一致

### 3.6 关键节点与关键边（双重 BFS）

### 必做版本：双重 BFS（朴素枚举）

- **时间复杂度**：**O(V·(V+E) + E·(V+E))**，即 **O((V+E)²)**
- **空间复杂度**：O(V+E)（每次 BFS 独立分配 visited 和队列，无持久化缓存）
- **实现要点**：

#### 图的方向性

本算法**仅支持无向图**。若 `IsDirected()` 为真，直接抛出异常。割点和桥的定义建立在无向连通图上，有向图的强连通分量分析需要完全不同的算法框架。

#### 调用接口

| 方法 | 说明 |
|------|------|
| `GetConnectedComponents` | 获取原图（全部 open 边）的连通分量数作为 baseline |
| `IsDirected` | `方向性处理` |
| `AllPlaceIds` | `收集所有待测试顶点` |
| `AllEdges` | `收集所有 status=="open" 的待测试边` |
| `GetAdjacentEdges` | `在 BFS 中动态获取邻接边，通过条件跳过被屏蔽的顶点或边` |

#### 模拟删除顶点：`blocked` 集合屏蔽

**机制**  
用 `unordered_set<string> blocked` 存放被删除的顶点。在 BFS 遍历时，从两个入口拦截：
1. **起点拦截**：若遍历到的起点在 `blocked` 中，直接 `continue`，不以它为起点启动 BFS。
2. **邻接拦截**：在遍历某顶点的邻接边时，若邻居 `v` 在 `blocked` 中，跳过该邻居，不入队。

**代码片段**
```cpp
if (blocked.count(start)) continue;          // 不在屏蔽顶点上启动 BFS
// ...
for (const auto &e : g.GetAdjacentEdges(u)) {
    const string &v = e.to_id;
    if (blocked.count(v)) continue;          // 不可达被屏蔽的邻居
    if (!visited.count(v)) { visited.insert(v); q.push(v); }
}
```

**时间复杂度**  
每次检查 `blocked.count()` 是 O(1) 的哈希查找。对一个顶点进行完整 BFS 的开销为 O(V+E)，与正常 BFS 一致，无额外遍历成本。

---

#### 模拟删除边：类型安全的 `pair` 比对

**机制**  
用 `pair<string,string>` 规范化边为 `(min_id, max_id)`，在 BFS 遍历时动态构造当前边的 pair 并与目标闭边比对。相比字符串拼接（`u+v`），`pair` 在类型层面消除歧义——不存在 `"ab|"+"|"+"c"` 和 `"ab"+"|"+"|c"` 这类隐藏冲突。`pair` 自带 `operator==`，无需额外比较逻辑。

**时间复杂度**  
每次边比对构造一个 `pair`（两个 string 拷贝，O(ID长度)），然后 `operator==` 逐字段比较，开销等同于字符串拼接方案，可视为 O(1)。

#### *设计细节：为何不用“逐边关闭”模拟顶点删除？*

| 对比维度 | 屏蔽顶点（当前实现） | 逐边关闭（不可取） |
|----------|---------------------|---------------------|
| **正确性** | 顶点从分量中彻底消失，等价于真删除 | 顶点本身还在，会被 BFS 当作孤立点单独计为一个分量，导致连通分量数虚高 |
| **效率** | 一次 `blocked.count()` O(1) | 需遍历 `deg(v)` 条边分别屏蔽，稠密图中开销成倍放大 |
| **实现** | 逻辑清晰，与边的屏蔽函数职责分离 | 混淆两类模拟，增加维护成本 |

*因此，当前实现严格采用“屏蔽顶点”处理关键节点、“规范化键比对”处理关键边，两者职责分明、互不混淆。*

#### 算法实现步骤
1. 调用 `GetConnectedComponents(graph)` 获得原图（全部 `open` 边）的连通分量数 `base_count`，作为后续比较的基准。
2. 通过 `AllPlaceIds()` 获取全部顶点列表，通过 `AllEdges()` 获取全部 `open` 边列表。
3. 对每个顶点 `v`，构造屏蔽集 `blocked = {v}`，调用 `CountCCWithBlocked(graph, blocked)` 计算删除该顶点后的连通分量数 `new_count`。若 `new_count > base_count`，则将 `v` 加入 `critical_nodes`。
4. 对每条 `open` 边 `e`，规范化其键 `key = min(from_id,to_id) + max(from_id,to_id)`，调用 `CountCCWithClosedEdge(graph, key)` 计算关闭该边后的连通分量数 `new_count`。若 `new_count > base_count`，则将 `(from_id, to_id)` 加入 `critical_edges`。
5. 封装 `CriticalResult` 并返回，其中 `critical_nodes` 为所有割点，`critical_edges` 为所有桥。
   
  
#### 局限性
这个 O((V+E)²) 的算法在面对**稠密图**和**大规模稀疏图**时存在严重效率瓶颈：

- **完全图 K₈₀**（80 个顶点，3160 条边）：关键边部分需要执行 3160 次 BFS，每次 O(3240)，总操作量约 1 千万次，实测耗时 **2.07 秒**。
- **网格图**（顶点数 1000 级别，边数约 2000）：暴力法运行 **2.06 秒**。

虽然对于校园图仍可高速完成，但对于特殊图其平方级劣势会被无限放大，所以需要引入 `Tarjan` 算法。

### 3.7 算法复杂度汇总

| 算法 | 时间复杂度 | 空间复杂度 | 实现方式 |
|------|----------|----------|---------|
| COMPONENTS | O(V+E) | O(V) | BFS + queue |
| SHORTEST | O((V+E) log V) | O(V) | Dijkstra + 小根堆 |
| TIMED_SHORTEST | O((V+E) log V) | O(V) | Dijkstra + 时间过滤 |
| MUST_PASS | O(k·(V+E) log V) | O(V) | k 段 Dijkstra 拼接 |
| MST | O((V+E) log V) | O(E) | Prim + 小根堆 |
| CRITICAL（朴素） | O(V·(V+E)+E·(V+E)) | O(V) | 逐点/逐边 BFS |
| CRITICAL（Tarjan） | O(V+E) | O(V) | 见拓展 |
| SHORTEST_K | O(K·(V+E)·log(KV)) | O(KV) | 见拓展 |

| # | 问题简述 | 解决方案 |
|---|----------|----------|
| 1 | `Dijkstra` 算法忽略惰性删除却同时却没有维护visited集合，算法被彻底搅浑 | 正确使用惰性删除：`if (d != dist[u]) continue;` |
| 2 | 分段 `Dijkstra` 维护路径时直接追加导致重复顶点 | 分类讨论：第一段直接追加全部节点，后续段跳过首元素（`seg.path.begin()+1`）避免与前一段终点重复 |
| 3 | `Dijkstra` 算法忽略提前终止，导致特定情况下分段 `Dijkstra` 龟速运行 | 增加提前终止逻辑 |
| 4 | `Tarjan` ：子节点回溯时未区分 parent 是否为 DFS 根，导致根节点被误判为割点 | `stack.back().parent == -1` 判断 `parent_is_root`，跳过非根判定 |

### 前端与通信问题

| # | 问题简述 | 解决方案 |
|---|----------|----------|
| 1 | 边不渲染：前端缓存 POST 响应，API 请求未到达 Flask 后端 | 启动时预生成 `graph_data.js`，前端 `<script>` 加载，零 HTTP |
| 2 | `static_url_path=""` 产生通配路由，拦截所有 `/api/*` 请求 | 删除 `static_url_path=""` |
| 3 | `LOAD` 命令路径被多余双引号包裹，`istringstream>>` 解析失败 | `LOAD "path"` → `LOAD path` |
| 4 | 多个旧 Python 进程同时监听端口，新进程无法正常接收请求 | `taskkill` 杀掉旧进程，固定端口 8080 |
| 5 | 浏览器未连接到 Flask（被旧进程抢端口），终端无任何请求日志 | 同上，杀旧进程后重启 |
| 6 | 响应格式从 `{edges}` 变为 `{mst: {edges}}`，前端未同步更新 | 前端适配 `data.mst.edges` |
| 7 | 非法数据在前端与 C++ 端行为不一致（前者跳过坏边，后者整批拒绝） | Web 标红非法边 + 禁用算法按钮 |
| 8 | MST 高亮不生效、不弹提示，因前端解析路径与后端不一致 | 同 #6，统一响应格式后修复 |
| 9 | 调试文件 `chk*.txt` 不生成——`collect_graph` 未执行（Edge 缓存了旧响应） | API 响应嵌入 `server_time` 字段确认新旧代码，定位到缓存问题 |
| 10 | 加载 `illegal_data` 后再也无法切换到其他数据集 | 禁用按钮的选择器 `button:not(.danger)` 误伤了"加载数据"按钮 → 改为 `.algo-btn` 类名精准定位 |

### 运行环境问题

| # | 问题简述 | 解决方案 |
|---|----------|----------|
| 1 | Windows Defender 与 McAfee 两种实时扫描导致程序启动耗时异常 | 将 `build/` 目录加入杀毒软件排除列表 |
