# 对抗样例

## 三、Confrontation — 对抗样例

`本栏目涉及四个极大的数据组，意图测试数据在极端条件下的运行情况`

### 3.1 complete_graph_K100

| 属性 | 值 |
|------|-----|
| 拓扑 | 完全图 K₁₀₀ |
| V / E | 100 / 4,950 |
| 主要测试 | CRITICAL 最坏情况 |

**特殊性**：完全图是朴素 CRITICAL 的死穴——100 + 4,950 = 5,050 次 BFS，每次遍历 ~5,050 个元素。且结果永远是 `NODES 0 EDGES 0` ，花费大量的时间结果只能算出空

**构造来源**：尽管现实里几乎没有与之匹配的数据结构，但这种建模的设计可以对双遍历算法造成极大压力，暴露其弱点

**能卡住的实现**：朴素CRITICAL算法（双BFS平方级开销）、边表（边数过多难以处理）、Dijkstra 不含提前退出（完全图中目标节点通常仅 1~2 跳，无 `break` 将遍历全部 V 个节点做无用功）

**如何处理**：CRITICAL 用 Tarjan O(V+E) 一次 DFS；Dijkstra 弹出目标节点后立即 `if (u == to_id) break`，避免 O(V log V) 的无用扩展

### 3.2 mass_grid_K1500

| 属性 | 值 |
|------|-----|
| 拓扑 | 30×50 网格 |
| V / E | 1,500 / 2,920 |
| 主要测试 | CRITICAL + SHORTEST系列等 综合压力 |


**特殊性**：网格图拥有仅略逊于完全图的密度，在 1,500 个节点上串联几乎所有核心命令，每次行动都只能走到最近节点的设计将所有图算法一并压力，验证其健壮性

**构造来源**：这一数据结构灵感来源于像素游戏——上下左右四向操作，它不仅可以用于对游戏内地点进行建模，庞大的顶点数以及行动次数也对算法造成不小的压力

**能卡住的实现**：任何不使用小根堆优化的算法（如朴素Dijkstra，朴素MST）、朴素CRITICAL算法（双BFS平方级开销）

**如何处理**：任何能够使用小根堆的算法决不朴素排序，同时精心设计数据结构，防止查邻接的时候开销过大

### 3.3 Big

| 属性 | 值 |
|------|-----|
| 拓扑 | 5 个稠密簇，簇间桥接 |
| V / E | 150 / 1,766 |
| 主要测试 | 聚类拓扑下的 CRITICAL 正确性 |

**特殊性**：5 个簇 × 30 节点，簇内以 80% 频率连边，相邻簇间各 2 条桥接边。每对簇有 2 座桥，删掉一座还有另一座连接——因此 CRITICAL 输出 `EDGES 0`，考验算法能否正确区分"真正的割结构"与"冗余连接"

**构造来源**：这一数据结构灵感来源于高内聚、少联结的社区模型，它不仅体现了最常见的社区结构关联，也可以检验算法是否可以正确区分真正的割结构与冗余连接

**能卡住的实现**：错误实现的 CRITICAL 算法（看似有割桥实则没有）

**如何处理**：正确实现 CRITICAL 的伪删除逻辑，保证代码准确性

### 3.4 prim_killer_K2000

| 属性 | 值 |
|------|-----|
| 拓扑 | 完全图 K₂₀₀₀ |
| V / E | 2,000 / 1,999,000 |
| roads.csv | ~50 MB |
| 主要测试 | LOAD + Prim MST + 空间复杂度|

**特殊性**：完全图是 Prim 堆操作的最大吞吐测试。堆峰值约 100 万条跨割边，共执行 ~200 万次 push + 200 万次 pop。邻接表双向存储约 400 万条记录

**升级历程**：K₆₀₀ (0.5s) → K₁₂₀₀ (2.3s) → K₂₀₀₀ (~6s)。每次翻倍节点数，边数增长约 4 倍

**构造来源**：同样的，此数据结构没有真实照应，它仅仅用于对Prim MST做压力测试，此测试同样能卡死CRITICAL

**能卡住的实现**：邻接矩阵与边表（内存与操作问题）、无小根堆优化的 Prim、Kruskal 的映射开销、朴素 CRITICAL（平方级）；Dijkstra 不含提前退出时，完全图中目标仅 1~2 跳即达，却被迫遍历全部 V 节点

**如何处理**：MST 用 Prim + 小根堆 + 惰性删除；Dijkstra 弹出目标后立即 `break`，完全图中可省去 ~99% 的无用扩展；使用嵌套哈希表降低空间复杂度

### 3.5 mass_chain_K2500

| 属性 | 值 |
|------|-----|
| 拓扑 | 链式图（2500 节点串行连接） |
| V / E | 2,500 / 2,499 |
| 主要测试 | CRITICAL 在链上的割点/桥极限数量 |

**特殊性**：除首尾外 2498 个节点都是割点，2499 条边全是桥——这是割点/桥密度最高的拓扑。朴素 CRITICAL 需执行 2500 + 2499 = 4999 次 BFS，实测 **4.63s**

**构造来源**：牵强地说，此数据结构构建于少分叉、多途径的高速公路，但更重要的是，链是验证 CRITICAL 正确性的最简拓扑图——每个内部节点和每条边都是关键设施。若算法输出不足 2498 割点或 2499 桥，则实现有误

**能卡住的实现**：朴素 CRITICAL（平方级 BFS 遍历）；若在此数据上反复调用 MUST_PASS 且 Dijkstra 未做 `if (u == to_id) break` 提前退出，也会严重拖累——链中每个分段 Dijkstra 必须遍历完全部节点

**如何处理**：CRITICAL 用 Tarjan O(V+E) 替代朴素 BFS；Dijkstra 在弹出目标节点后立即 `break`，避免遍历整条链的剩余部分
